## Abstract

We present two tightly coupled contributions for instruction-guided music audio manipulation. **ProCraft-Data** is a large-scale dataset (~45K–90K entries after quality filtering) where each record contains an original multi-track audio mixture, a production motivation, a chain-of-thought thinking trace with programmatically executed tool calls (generated by Qwen3-30B-A3B-Thinking; Alibaba, 2025), and a verified modified audio output, covering 85 audio effects from the MultiAFX library plus instrument re-rendering via FluidSynth and MIDI-level performance modification. **ProCraft-Model** is a text-conditioned factorized codec that disentangles frozen DAC (Rithesh Kumar et al., NeurIPS 2023) tokens into three semantically named factors — **[TMB]** (timbre/instrument identity), **[EXP]** (expression: dynamics, effects character, articulation), and **content** (pitch/rhythm structure) — using periodic token injection inspired by ALMTokenizer (Dongchao Yang et al., 2025), an information bottleneck on content tokens supervising pianoroll prediction, gradient reversal layer disentanglement (Ganin & Lempitsky, ICML 2015), and a reconstruction-first curriculum. During text-conditioned training, the text instruction shifts which labels supervise the factor tokens and which audio serves as reconstruction target — the encoder learns to produce modified factor representations directly from original audio + text, requiring no reference audio at inference. On a held-out test of 500 ProCraft-Data pairs, ProCraft-Model is expected to improve content preservation rate by ≥10 F1 points over a monolithic ablation baseline while matching or exceeding it on perceptual quality (FAD).

## 1. Problem Statement and Motivation

Instruction-guided audio manipulation — changing the timbre of a lead piano to a warm Rhodes, adding tape saturation, or softening articulation — requires a model to understand *which acoustic property* the instruction targets and to modify that property *without disturbing the others*. Existing approaches are monolithic: they encode text and audio into a shared latent space and predict modified audio directly (AUDIT, Zhifang Wang et al., NeurIPS 2023; InstructME, IJCAI 2024; Instruct-MusicGen, Yixiao Zhang et al., ISMIR 2025). In a monolithic latent space, timbre, expression, and content are entangled, so the model has no structural mechanism to enforce factor-selective modification. A user instruction "change piano to guitar" should affect only timbre — not dynamics, not melodic content — but a monolithic model can only learn this statistically.

A second problem is training data: existing instruction-audio datasets either cover too few effects (LLM2Fx-Tools, Sony, Dec 2025: 9 effects, 101K pairs with chain-of-thought), omit reasoning traces entirely (AUDIT: no CoT), or target audio generation rather than editing (ThinkSound, NeurIPS 2025: Foley generation). None spans the breadth of production tools a human producer uses, and none captures the *reasoning process* behind a decision.

**Core insight:** Music production decisions follow a structured reasoning process (motivation → diagnosis → tool selection → parameter tuning) that a thinking language model naturally generates. Capturing this process in a dataset and training a *factorized* codec on the resulting paired audio yields a model that makes interpretable, factor-specific modifications with structural guarantees on content preservation.

## 3. Contribution 1 — ProCraft-Data Dataset

### 3.1 Source Material

We combine multiple multitrack MIDI sources to ensure sufficient scale and diversity:

- **Slakh2100** (Ethan Manilow et al., WASPAA 2019): 2,100 curated multi-track MIDI files rendered with professional VSTs. Per-track GM instrument labels, full MIDI ground truth. High quality but limited scale.
- **GigaMIDI** (Metacreation Lab, 2025): 2.1M MIDI files, ~922K multitrack (43.2%). All 128 GM instrument programs annotated. Expressive performance detection (velocity variation, onset deviation). We filter for multitrack files with ≥3 instrument tracks and expressivity score (NOMML) above threshold → estimated ~200K–400K usable files.
- **Lakh MIDI Dataset** (Raffel, 2016): 176K unique MIDI files, ~170K multitrack. CC BY 4.0 license (most permissive). Combined with **MidiCaps** annotations (168K files with tempo, chords, genre, mood, instrument labels) for rich metadata.

### 3.1.1 Audio Rendering Pipeline

All MIDI sources are rendered per-stem to **48 kHz** stereo audio. (Diverges from the original 44.1 kHz spec: 48 kHz matches standard video/DAW post-production rates and most modern VSTi presets, and defers any resampling cost to the DAC pre-processing step where it can be batched on GPU.) We employ a two-tier rendering strategy to balance fidelity and throughput at scale (100K+ files):

**Tier 1 — High-fidelity rendering (primary):**
- **DawDreamer** (DBraun, open source) + commercial VSTi instruments. DawDreamer is a Python-native headless DAW that loads VST2/3 instruments, accepts MIDI input, and renders to audio — with a built-in multiprocessing pipeline (N workers = CPU count, each with a persistent RenderEngine consuming from a shared Queue).
- **Recommended VSTi sources:**
  - **EastWest ComposerCloud+** (~$20–40/month subscription): hundreds of orchestral, pop, and world instruments at studio quality (quality: 8–9/10 vs real instruments)
  - **Native Instruments Kontakt** ($400, full version): industry-standard sampler with extensive third-party libraries
  - **Arturia V Collection** (~$400): vintage synthesizer emulations for electronic/pop timbres
- **Throughput:** Renders faster than real-time per worker. With 16 CPU cores, estimated ~2–4 days for 100K files.
- **Platform:** macOS, Windows, Linux, Docker (containerizable for cloud distribution via AWS Batch or RunPod).

**Tier 2 — Fast rendering for development and augmentation:**
- **FluidSynth** (open source) with curated GM soundfonts (Timbres of Heaven, SGM v2.01, GeneralUser GS, Sonatina Symphonic Orchestra, and 20+ from the 500 Soundfonts Collection on Internet Archive). Quality: 4–5/10, but trivially parallelizable — each Python `multiprocessing` worker runs an independent FluidSynth instance. 100K files renderable in hours on a single machine.
- Used for: initial pipeline development, rapid iteration, soundfont-diversity augmentation (same MIDI × multiple soundfonts).

**Alternative options (evaluated but not primary):**
- **Pedalboard** (Spotify, open source): VST3 instrument hosting with GIL-free threading (no multiprocessing needed). Strong alternative to DawDreamer but VST3-only (no VST2) and segfault reports at 8K+ files.
- **REAPER + ReaRender** ($60 license): Python-scriptable DAW with native multi-core batch rendering. Proven at scale (9K+ MIDI files). More setup overhead than DawDreamer.
- **Neural synthesis (MIDI-DDSP, MIAO)**: Higher per-note realism (6–7/10) but GPU-bound inference is orders of magnitude slower than sample-based rendering. Suitable as a complement for specific solo instruments, not for full-scale dataset generation.

Each track is rendered separately then mixed with ITU-R BS.1770-4 loudness normalization, providing per-stem isolation for track-level operations (instrument change, effects, removal).

**Data augmentation during generation:**
- **Timbral diversity:** Each MIDI file rendered with 3–5 different soundfonts (Tier 2) or VSTi presets (Tier 1)
- **Tempo variation:** ±10% tempo scaling on a subset of files
- **Transposition:** ±{1,2,3} semitone shifts on a subset
- **Track subsetting:** Random subsets of tracks from files with >4 instruments, creating mixtures of varying complexity

This yields a source pool of **~50K–100K unique multitrack audio mixtures** (after quality filtering), substantially larger than Slakh2100 alone.

### 3.2 Generation Pipeline

Each dataset entry = (original_audio, production_motivation, thinking_trace, tool_calls, modified_audio).

**Step 1 — Audio description extraction.** From Slakh metadata: instrument label per track, tempo (BPM), key signature, note density (notes/sec per track), velocity range per track, duration.

**Step 2 — Motivation and trace generation via Qwen3-30B-A3B-Thinking.** We query **Qwen3-30B-A3B-Thinking** (30.5B total parameters, 3.3B active per token via Mixture-of-Experts with 128 experts / 8 active; 256K context window; Alibaba, 2025). This model natively emits `<think>...</think>` chain-of-thought before producing structured outputs. The system prompt defines all available tools in Hermes-style `<tools>` XML with full JSON parameter schemas.

Example system prompt excerpt:
```xml
<system>
You are a professional music producer. You have access to these tools:
<tools>
[{"name": "change_instrument", "description": "Re-render a MIDI track with a different soundfont preset",
  "parameters": {"track": {"type": "string"}, "from_preset": {"type": "string"}, "to_preset": {"type": "string"}}},
 {"name": "apply_fx", "description": "Apply an audio effect to a track or the mix",
  "parameters": {"effect": {"type": "string", "enum": ["tape_saturation","parametric_eq",...]},
                 "params": {"type": "object"}, "track": {"type": "string", "default": "mix"}}},
 ...]
</tools>
Generate a specific production motivation, reason about which tools achieve it, then output tool calls.
</system>
```

Example model output:
```xml
<think>
This mixture has a late-night jazz feel. The acoustic piano is too bright for the mood.
Replacing with a Rhodes electric piano and adding subtle tape saturation will add warmth.
The bass could use light compression to sit better under the melody.
</think>

Production motivation: Create a late-night jazz atmosphere by warming the piano timbre
and tightening the low end.

<tool_call>
{"name": "change_instrument", "arguments": {"track": "piano", "from_preset": "acoustic_grand", "to_preset": "rhodes_ep"}}
</tool_call>
<tool_call>
{"name": "apply_fx", "arguments": {"effect": "tape_saturation", "params": {"drive": 0.3, "warmth": 0.6}, "track": "piano"}}
</tool_call>
<tool_call>
{"name": "apply_fx", "arguments": {"effect": "compressor", "params": {"threshold_db": -18.0, "ratio": 3.0, "attack_ms": 10.0, "release_ms": 80.0}, "track": "bass"}}
</tool_call>
```

**Step 3 — Tool execution.** Four tool categories covering the full breadth of music production, all executed programmatically:

**Category A — Instrument Changes (re-rendering via DawDreamer or FluidSynth):**
- `change_instrument(track, from_preset, to_preset)` — re-renders the track's MIDI with a different instrument preset (Tier 1: VSTi preset via DawDreamer; Tier 2: GM soundfont preset via FluidSynth)
- `layer_instrument(track, additional_preset, mix_ratio)` — adds a second instrument layer at specified blend ratio

**Category B — Audio Effects (MultiAFX library, 85 effects):**
Full parameter-specified effects across: EQ (parametric, shelving, graphic); Dynamics (compressor, limiter, gate, expander, transient shaper); Time-based (reverb, delay, chorus, flanger, phaser); Distortion (overdrive, tape saturation, bitcrusher, fuzz); Modulation (tremolo, vibrato, ring mod, auto-wah); Spatial (stereo widener, panner, mid-side processing); Spectral (vocoder, pitch shifter, formant shifter). Where MultiAFX has not yet been released, effects are implemented using pedalboard (Spotify) and librosa/scipy as fallback.

**Category C — Performance Modifications (pretty_midi + FluidSynth):**
- `change_velocity(track, scale_factor)` — scales MIDI velocity uniformly (dynamics modification)
- `humanize_timing(track, max_offset_ms)` — adds random microtiming offsets within ±max_offset_ms
- `change_articulation(track, style)` — modifies note durations (legato: +20% duration; staccato: −40% duration)

**Category D — Track Addition / Removal (arrangement changes):**
These operations modify the content of the mixture, making ProCraft-Data a general music production dataset rather than a timbre-only dataset.
- `add_track(track_name, preset, source)` — adds a track that exists in the original MIDI but was excluded from the "original audio" input. Implementation: the original mixture is rendered *without* this track; the tool call adds it back. This creates a natural (input=incomplete mix, output=complete mix) pair.
- `remove_track(track_name)` — removes a track from the mixture. Implementation: the modified audio simply omits that track's stem. Creates (input=full mix, output=mix minus one track).
- `double_track(track_name, offset_ms, detune_cents)` — duplicates a track with slight timing/pitch offset for a thicker sound
- `mute_and_replace(track_name, new_preset)` — removes an existing track and adds a new one playing the same MIDI with a different instrument (combines remove + add)

**Production diversity through combined operations:** The thinking model naturally combines tools from all four categories in a single production decision. For example: "The arrangement feels thin — add a string pad, apply reverb to it, and reduce the piano velocity to make room" involves `add_track` + `apply_fx` + `change_velocity`.

**Content modification handling:** Unlike timbre/effects-only changes, Category D operations (add/remove tracks) change the musical content of the mixture. The dataset records which category each tool call belongs to, enabling the downstream model to adjust supervision targets accordingly (see Section 4.6).

**Step 4 — Quality filtering.** Discard entries where:
- FAD (Fréchet Audio Distance, VGGish features) between original and modified mix exceeds 2× median FAD across all modifications in the batch (too destructive)
- FAD between original and modified is below 0.05 (modification inaudible)
- Tool execution fails (FluidSynth render error, invalid parameter range, audio clipping > 0.1% of samples)

**Step 5 — Diversity sampling (25–50 motivations per track).** To prevent duplicate motivations, we vary: generation temperature (0.7, 0.9, 1.1), style target appended to user prompt ("for a film score", "for a lo-fi hip-hop playlist", "for a theatrical performance", "for a commercial jingle", "for a video game soundtrack"), and number of tool calls requested (1, 2, 3, or 4). This structured variation yields diverse motivations while keeping generation cost tractable.

**Compute for generation:** Qwen3-30B-A3B-Thinking with vLLM batching on 2× A100-80GB, batch size 16: estimated 48 hours for 105K generation calls (before filtering), resulting in ~45K–90K final entries.

## 4. Contribution 2 — ProCraft-Model Architecture

### 4.1 Design Overview

ProCraft-Model operates entirely on tokens from a frozen **DAC** (Rithesh Kumar et al., NeurIPS 2023) codec. The DAC encoder maps waveforms to RVQ token embeddings; the DAC decoder maps predicted RVQ token indices back to waveforms. ProCraft-Model learns the intermediate transformation: given (optional) text instruction and original DAC tokens, produce new DAC tokens corresponding to the modified audio.

The three factors are semantically named to reflect what they capture:
- **[TMB]** (Timbre) — instrument identity, spectral envelope, attack/decay character
- **[EXP]** (Expression) — dynamics contour, effects character (reverb tail, saturation coloring), articulation style; operationally defined as everything not captured by [TMB] or content
- **Content (c)** — pitch and rhythm structure: which notes are played and when

### 4.2 Frozen DAC Tokenizer

Dataset audio is stored at 48 kHz, 30 seconds per clip. Because the public DAC checkpoint is 44.1 kHz-native, audio is resampled 48 → 44.1 kHz on the fly immediately before DAC encoding (GPU-side with `torchaudio.functional.resample`, so no extra disk cost). The frozen DAC encoder then maps a 30-second clip to:
```
z = [z_1, z_2, ..., z_T]    T ≈ 2580 frames (44.1 kHz / 512-sample hop × 30 s)
```
Each z_t is a sum of N_q = 12 RVQ codebook embeddings (d_model = 1024). The DAC encoder and decoder weights are frozen throughout all training.

### 4.3 Text Encoder

Text instructions (serialized tool call strings, e.g., `"change_instrument: acoustic_grand → rhodes_ep | apply_fx: tape_saturation drive=0.3"`) are encoded by a frozen **T5-small** encoder (60M parameters; Raffel et al., 2020). T5-small is chosen over FLAN-T5 because: (1) instruction-following is not needed — we need token-level semantic embeddings of syntactically regular strings; (2) T5-small fits comfortably alongside the rest of the pipeline in an A100-80GB GPU's memory budget; (3) the tool-call vocabulary is domain-specific and does not require the instruction-tuning that FLAN-T5 provides.

T5-small encodes up to 64 tokens → embeddings `[e_1, ..., e_M]` (dim 512). A two-layer MLP adapter projects to d_model = 1024 (matching DAC embedding dimension).

### 4.4 Unified Encoder with Periodic Factor Token Injection

Inspired by the periodic token injection pattern in **ALMTokenizer** (Dongchao Yang et al., 2025) — which inserts special tokens at fixed intervals into codec token streams to aggregate local context at multiple timescales — we inject two learnable special tokens every w = 8 DAC frames:

```
Encoder input sequence:
[e_1, ..., e_M,   z̃_1...z̃_8, [TMB]_1, [EXP]_1,   z̃_9...z̃_16, [TMB]_2, [EXP]_2,   ...]
 \_text prefix_/  \_window 1 + special tokens_/     \_window 2 + special tokens_/
```

where `z̃_t` denotes a DAC token with ~20% random masking applied (masked tokens replaced with a learned [MASK] embedding). The 20% masking rate follows MAE (He et al., CVPR 2022) — light enough to preserve local structure, sufficient to force special tokens to aggregate information beyond local context. Text embeddings are prepended only during text-conditioning training (absent during reconstruction-only training; see curriculum, Section 4.7).

**N = 12 transformer self-attention layers** with full bidirectional attention process the entire sequence. After encoding, factor representations are extracted by position type:
```
t_tmb = {outputs at [TMB]_1, [TMB]_2, ..., [TMB]_K}    shape: [K, d_model], K = T/w
t_exp  = {outputs at [EXP]_1, [EXP]_2, ..., [EXP]_K}    shape: [K, d_model]
c      = {outputs at z positions}                          shape: [T, d_model]
```

### 4.5 Disentanglement Supervision

All three factors receive dedicated supervision at every training step. The text instruction changes *which audio provides the target factor values* — it does not gate whether supervision is applied. This "always-supervised" design prevents any factor from going dark during text-free reconstruction steps.

**[TMB] — Instrument classification + GRL against pianoroll (always active):**  
A linear head on each [TMB]_k predicts which General MIDI instruments are active in window k (multi-label binary cross-entropy against Slakh instrument labels). Additionally, a Gradient Reversal Layer connects [TMB] tokens to the pianoroll predictor (λ_tmb_content = 0.1), ensuring timbre tokens do NOT encode pitch/rhythm content — timbre should be content-invariant. Loss: L_tmb_cls + L_grl_tmb.

**Content — Pianoroll probe + GRL (always active):**  
Content tokens `c` pass through an **information bottleneck**: a linear projection to 8 dimensions, which then feeds a pianoroll prediction head (88-pitch × w-frame binary cross-entropy against MIDI ground truth). The 8-dim bottleneck is on the **main content pathway** — this is intentional. By forcing content through a narrow bottleneck that must predict pianoroll, we ensure content tokens encode ONLY pitch/rhythm structure and discard all timbre and expression information. This bottleneck is the primary structural mechanism for content disentanglement — it limits content capacity to exactly what pianoroll supervision requires. Additionally, a Gradient Reversal Layer (GRL; Ganin & Lempitsky, ICML 2015) connects content tokens to the instrument classifier, discouraging content tokens from encoding instrument identity. GRL gradient weight: λ_c = 0.1. Loss: L_content_piano + L_grl_c.

**[EXP] — Dual GRL (always active):**  
[EXP] tokens connect via GRL to *both* the instrument classifier (λ_exp_tmb = 0.1) *and* the pianoroll predictor (λ_exp_content = 0.1). Information that survives both adversarial pressures — captured neither by instrument identity nor by pitch/rhythm — is by exclusion the expression factor. No separate positive supervision target for [EXP] is required. Loss: L_grl_exp (two GRL branches).

**GRL warmup schedule:** Both λ values increase linearly from 0 to target over the first 30K steps, following the standard ramp used in FACodec (Zeqian Ju et al., 2024) and the original GRL paper (Ganin & Lempitsky, ICML 2015), to prevent gradient instability at initialization.

**FACodec-inspired perturbation training:**  
- *Timbre perturbation for content invariance:* Randomly pitch-shift original audio by ±{1, 2} semitones before DAC encoding. The perturbed audio shares the same instrument (same t_tmb) but different content. A consistency loss penalizes L2 distance between t_tmb of original and pitch-shifted audio.
- *Content perturbation for timbre invariance:* Apply random segment shuffling to the MIDI pianoroll (permute 4-bar segments), re-render. Shuffled content differs; instrument labels (and thus t_tmb supervision) remain identical.

These perturbations, inspired by FACodec (Zeqian Ju et al., 2024), explicitly train factor invariance that the GRL alone might not achieve.

### 4.6 Text-Conditioned Training Design (No Factor Swapping)

A key design principle: we do NOT encode a separate reference/modified audio to obtain target factor tokens, and there is no "factor generator" or "delta prediction" module. Instead, the text instruction shifts what the encoder's own factor tokens should represent, and the reconstruction target is the modified audio.

**Core principle:** We always feed original audio + text instruction to the encoder. The text instruction determines (a) which labels supervise the factor tokens and (b) what audio serves as the reconstruction target. No reference/modified audio is encoded separately — the encoder learns to shift its own factor tokens based on the text signal.

**Important: timbre labels are mixture-level.** Since we do not extract single sources, the [TMB] label is the full instrument composition of the mixture. For example, if the original mixture contains {piano, bass, drums} and the instruction swaps piano to violin, the TARGET timbre label is {violin, bass, drums} — not just "violin."

**Example 1 — Instrument change ("swap piano to violin"):**
1. Feed original audio (piano+bass+drums) + text to encoder
2. [TMB] supervised with **target** mixture label: {violin, bass, drums}
3. Content supervised with **original** pianoroll (same MIDI notes)
4. Reconstruction target: modified audio (violin+bass+drums)

**Example 2 — Effects change ("add tape saturation to piano"):**
1. Feed original audio + text to encoder
2. [TMB] supervised with **modified** timbre label — the instruments are the same {piano, bass, drums} but the timbral character has changed (saturation alters spectral content), so [TMB] must capture this shift via the reconstruction loss. The instrument classification label stays the same, but the reconstruction target has different timbre.
3. Content supervised with **original** pianoroll
4. Reconstruction target: modified audio (with saturation applied)

**Example 3 — Track addition ("add string pad"):**
1. Feed original audio (piano+bass+drums — string pad track withheld) + text to encoder
2. [TMB] supervised with **target** mixture label: {piano, bass, drums, strings}
3. Content supervised with **target** pianoroll (original MIDI + string pad MIDI)
4. Reconstruction target: modified audio (piano+bass+drums+strings)

**Example 4 — Track removal ("remove drums"):**
1. Feed original audio (piano+bass+drums) + text to encoder
2. [TMB] supervised with **target** mixture label: {piano, bass}
3. Content supervised with **target** pianoroll (original MIDI minus drum track)
4. Reconstruction target: modified audio (piano+bass only)

**Key insight:** For content-modifying operations (add/remove track), the content tokens are supervised with the **modified** pianoroll, not the original. This is identical in principle to how [TMB] is supervised with modified labels — whatever the text instruction says should change, the corresponding factor's supervision target changes accordingly.

**Consistency between training and inference:** At both training and inference, the model receives the same inputs: text instruction + original audio. At training, we have modified audio/labels as targets. At inference, the encoder has learned to shift its factor tokens based on text — no reference audio encoding needed.

**Inference requirement — instrument composition metadata:** For the thinking model (Qwen3) to generate valid tool calls at inference, it must know the instrument composition of the input audio (e.g., "tracks: piano, bass, drums"). This same metadata is included in the text prompt to ProCraft-Model for consistency. During training, this comes from Slakh metadata. At inference on unseen audio, an instrument classifier (e.g., the [TMB] classifier itself) provides this information.

### 4.7 Decoder

For each window k of w = 8 frames:
```
decoder_input_k = concat([t_tmb_k, t_exp_k, c_{(k-1)·w+1}, ..., c_{k·w}])
```

**M = 8 transformer decoder layers** with cross-attention (content positions query factor tokens) followed by a linear prediction head producing 12 × 1024-way softmax over RVQ token indices (12 RVQ levels, codebook size 1024). Predicted indices are passed to the frozen DAC decoder to produce audio.

**Decoder input table (P1 — Information Sufficiency):**

In ALL modes, the encoder processes original audio (with optional text prepended). The encoder's own output tokens go directly to the decoder — no factor swapping or delta prediction.

| Slot | Training (text: instrument change) | Training (text: effects/perf change) | Training (recon: no text) | Inference |
|------|------------------------------------------|--------------------------------------------|---------------------------------|-----------|
| t_tmb_k | Encoder [TMB] from original audio, supervised with **target** inst. label | Encoder [TMB] from original audio, supervised with **same** inst. label | Encoder [TMB], supervised with inst. label | Encoder [TMB] (text shifts representation) |
| t_exp_k | Encoder [EXP] from original audio, dual GRL | Encoder [EXP] from original audio, dual GRL | Encoder [EXP], dual GRL | Encoder [EXP] (text shifts representation) |
| c_{...} | Content from original audio, bottleneck → pianoroll | Content from original audio, bottleneck → pianoroll | Content from original audio, bottleneck → pianoroll | Content from original audio |
| Recon target | **Modified audio** (instrument swapped) | **Modified audio** (effects applied) | **Original audio** | N/A (generate output) |
| Text input | "change_instrument: piano → violin" | "apply_fx: tape_saturation ..." | None | User instruction |

### 4.8 Curriculum Training Schedule

Total: **300K steps**, single A100-80GB, batch size 32, BF16 mixed precision. Estimated wall time: ~96 hours (4 days).

| Phase | Steps | Text prompt probability | Primary focus |
|-------|-------|------------------------|--------------|
| 1 — Disentanglement | 0–90K (30%) | 0% | Reconstruction + disentanglement; GRL warmup 0→0.1 |
| 2 — Ramp | 90K–150K (20%) | 0% → 50% linear | Both reconstruction and text-conditioning introduced |
| 3 — Joint | 150K–240K (30%) | 50% | Full joint training; factor generator fully active |
| 4 — Consolidation | 240K–300K (20%) | 80% | Emphasize text conditioning; GRL λ decayed by 0.5× |

**Rationale:** Training reconstruction-first follows the approach of multimodal models that learn visual representations before language alignment. Early text conditioning would cause the model to exploit text as a shortcut, bypassing disentanglement learning.

### 4.9 Training Mode Loss Table

| Mode | Encoder input | [TMB] label | Content label (pianoroll) | [EXP] | Recon target |
|------|--------------|-------------|--------------------------|-------|-------------|
| Recon (no text) | z̃ (masked DAC) | Source inst. composition | Source pianoroll | Dual GRL | Original audio |
| Text: instrument change | [text_emb] + z̃ | **Modified** inst. composition | Source pianoroll | Dual GRL | Modified audio |
| Text: effects change | [text_emb] + z̃ | **Modified** (timbre changed by effects) | Source pianoroll | Dual GRL | Modified audio |
| Text: performance change | [text_emb] + z̃ | Source inst. composition | Source pianoroll | Dual GRL | Modified audio |
| Text: add track | [text_emb] + z̃ | **Modified** inst. composition | **Modified** pianoroll | Dual GRL | Modified audio |
| Text: remove track | [text_emb] + z̃ | **Modified** inst. composition | **Modified** pianoroll | Dual GRL | Modified audio |

Note: ALL supervision losses are ALWAYS active in ALL modes. The text instruction determines which labels change (source → modified) and which stay the same. Timbre labels are always mixture-level (e.g., {violin, bass, drums}), not single-instrument.

### 4.10 Inference Pseudocode

```python
# Algorithm 1: ProCraft-Model Inference
# Input:  original_audio    — waveform (48 kHz stereo, 30 sec; resampled to 44.1 kHz before DAC)
#         text_instruction  — string, e.g. "change_instrument: acoustic_grand → rhodes_ep"
#         instrument_metadata — string, e.g. "tracks: piano, bass, drums"
# Output: modified_audio    — waveform

def factorfx_inference(original_audio, text_instruction, instrument_metadata):
    # Step 1: DAC encode (frozen) — expects 44.1 kHz input
    audio_44k = resample(original_audio, 48000, 44100)
    z_orig    = dac_encoder(audio_44k)                # [T=860, d_model=1024]

    # Step 2: Text encode + project
    # Prepend instrument metadata for consistency with CoT generation
    full_text = f"{instrument_metadata} | {text_instruction}"
    text_tokens = t5_encoder(full_text)               # [M≤64, 512]
    text_emb    = mlp_adapter(text_tokens)            # [M, 1024]

    # Step 3: Unified encoder (no masking at inference)
    # Build: [text_emb | z_1..z_8 [TMB]_1 [EXP]_1 | z_9..z_16 [TMB]_2 [EXP]_2 | ...]
    seq     = build_sequence(text_emb, z_orig, w=8)   # [M + T + 2K, 1024]
    encoded = unified_encoder(seq)                     # 12-layer transformer

    # Step 4: Retrieve factor tokens by position
    t_tmb = encoded[[TMB] positions]                   # [K, 1024]
    t_exp = encoded[[EXP] positions]                   # [K, 1024]
    c     = encoded[z positions]                       # [T, 8] (after bottleneck)

    # Step 5: Decode — the encoder has already shifted [TMB]/[EXP]
    # tokens based on the text instruction (learned during training)
    rvq_logits = []
    for k in range(K):
        window = concat([t_tmb[k],                    # [1024]
                         t_exp[k],                    # [1024]
                         c[k*w : (k+1)*w]])           # [w=8, 8]
        rvq_logits.append(decoder(window))          # [w, 12, 1024]
    rvq_tokens = argmax(concat(rvq_logits), dim=-1) # [T, 12]

    # Step 7: DAC decode (frozen)
    modified_audio = dac_decoder(rvq_tokens)        # waveform

    return modified_audio
```

## 5. Baselines

1. **No-Disentanglement (primary ablation):** Monolithic transformer with same capacity (N=12 encoder layers, same frozen DAC tokenizer, same T5-small text encoder). Text and audio are concatenated and processed jointly; decoder directly predicts modified DAC tokens. No [TMB], no [EXP], no GRL, no factor generator, no information bottleneck. This isolates the contribution of factorized disentanglement.

2. **Instruct-MusicGen** (Yixiao Zhang et al., ISMIR 2025): Template-based instruction tuning for stem editing. Adapted to same test pairs.

3. **AUDIT** (Zhifang Wang et al., NeurIPS 2023): Instruction-guided audio editing without reasoning traces or factorized representation.

4. **InstructME** (IJCAI 2024): Text-conditioned music editing with monolithic latent space.

**Note on LP-Fx:** LP-Fx is a prior dataset/system that we cite for context but do not use as a baseline — it targets a different task formulation and is not a model we can run on our test set. **NSynth is not used** (single isolated notes, not multi-track mixtures).

## 6. Evaluation

### 6.1 Test Sets

- **In-distribution test:** 210 held-out Slakh2100 tracks (10% split by track) × 5 motivations = 1,050 pairs
- **Out-of-distribution test:** 100 pairs from MUSDB18-HQ (real recordings, not MIDI-rendered, different acoustic conditions)

### 6.2 Metrics

| Metric | What it measures | Computation |
|--------|-----------------|-------------|
| FAD (Fréchet Audio Distance) | Perceptual audio quality | VGGish features; lower = better |
| KAD (Kernel Audio Distance) | Complementary perceptual quality metric | MMD-based kernel distance on audio embeddings; lower = better |
| Content Preservation Rate (CPR) | Pitch/rhythm preservation after modification | Piano-roll F1 between generated and target MIDI, ±50ms onset tolerance; higher = better |
| Instrument Classification Accuracy (ICA) | Timbre factor coherence | Pre-trained classifier accuracy on modified output; higher = better |
| Effects Fidelity (EF) | Accuracy of applied effects | Parametric distance between target and generated effect parameters (where ground truth available); lower = better |
| MOS (Mean Opinion Score) | Overall perceptual quality | 5-point scale, 20 listeners × 50 pairs (supplementary) |

### 6.3 Ablation Studies

1. No GRL on content (pianoroll bottleneck only, no adversarial pressure on c)
2. No dual-GRL on [EXP] (only instrument GRL, no pianoroll GRL)
3. No GRL on [TMB] against pianoroll (instrument classification only)
4. No information bottleneck (full d_model content, pianoroll probe-only)
5. No FACodec perturbation (no pitch-shift / MIDI shuffle)
6. Full ProCraft-Model (all components active)

## 7. Minimal Experiment (Go/No-Go Criterion)

Train ProCraft-Model on 5,000 pairs (≈10% of estimated corpus) for 50,000 steps on a single A100-80GB. Thresholds for proceeding to full training:

| Metric | Threshold | Expected baseline (No-Disentanglement) |
|--------|-----------|--------------------------------------|
| Content Preservation Rate (CPR) | ≥ 0.80 F1 | ~0.65 |
| Instrument Classification Accuracy (ICA) | ≥ 0.75 | ~0.55 (random factors) |
| FAD vs. reference | ≤ 2× No-Disentanglement FAD | baseline |

If CPR < 0.80, investigate whether pianoroll probe gradients are interfering with reconstruction (check L_recon curve separately; if it plateaus early, reduce probe loss weight by 0.5×).

## 8. Why It Could Work

**ProCraft-Data feasibility:** Qwen3-30B-A3B-Thinking has demonstrated native tool-calling in Hermes XML format. Slakh2100 provides the clean MIDI and instrument labels needed for programmatic execution and supervision. The content preservation constraint is trivially enforceable (no tool modifies MIDI pitches or onsets). FAD-based filtering is standard in audio generation literature.

**ProCraft-Model feasibility:**
- Periodic token injection is validated in ALMTokenizer (Dongchao Yang et al., 2025) for audio language modeling.
- GRL-based factor disentanglement is validated in FACodec (Zeqian Ju et al., 2024) for speech (timbre/content/acoustic detail) and in the foundational GRL work (Ganin & Lempitsky, ICML 2015) for domain adaptation.
- The information bottleneck (8-dim on the main content pathway) is a strong structural constraint that forces content to encode only pianoroll-level structure, providing hard disentanglement guarantees.
- The text-conditioned training design (text shifts supervision labels + reconstruction target, encoder produces shifted factor tokens directly) is simpler and more principled than reference-based factor swapping — no separate encoding of modified audio needed during inference.
- Curriculum training mirrors standard multimodal practice (vision backbone trained before language alignment).

**Why factorized beats monolithic:** Structural content preservation (always use c from original) provides a hard guarantee that monolithic models can only approximate statistically. This architectural constraint should yield ≥10 F1 improvement in CPR at similar or better FAD.

## 9. Novelty Claims

**ProCraft-Data vs. LLM2Fx-Tools (Sony, Dec 2025):** 85+ effects vs. 9; genuine thinking traces from a thinking model vs. manually designed CoT templates; MIDI-level performance modification (velocity, articulation, humanization) absent from LLM2Fx-Tools; instrument re-rendering via FluidSynth absent from LLM2Fx-Tools; arrangement operations (add/remove tracks) for content-modifying production decisions; 400K–2M entries vs. 101K; multiple MIDI source pools (Slakh + GigaMIDI + LMD) with data augmentation.

**ProCraft-Model vs. DisMix (Yuang Luo et al., 2024):** Text-conditioned (DisMix is not); three factors including [EXP] (DisMix separates timbre and content only); trained on production modification pairs rather than stem separation pairs.

**ProCraft-Model vs. FACodec (Zeqian Ju et al., 2024):** Music domain not speech; three factors not two; text-conditioned with inference-time factor generation; trained on ProCraft-Data paired data.

**ProCraft-Model vs. Instruct-MusicGen (Yixiao Zhang et al., ISMIR 2025):** Factorized representation structurally enforces content preservation; explicit inference pseudocode with factor routing; 85-effect coverage from ProCraft-Data training data.

## 10. Risks and Mitigations

| Risk | Severity | Mitigation |
|------|----------|-----------|
| Qwen3 generation throughput too slow for 100K entries | Medium | vLLM batching on 2× A100; reduce corpus to 20K if needed (still 2× LLM2Fx-Tools effects coverage) |
| GRL training instability — factors collapse to similar representations | High | GRL λ warmup (0→target over 30K steps); monitor mutual information (MINE probe) every 5K steps; if collapse detected, halve λ and extend warmup |
| Factor generator produces out-of-distribution deltas for unseen instructions | Medium | Norm clipping on Δt (max_norm=0.5); diverse training motivations via temperature/style variation |
| DAC token reconstruction degrades when combining factors from different clips | Medium | FACodec-style perturbation training explicitly trains cross-clip factor combination |
| MultiAFX library not yet released | Medium | Supplement with pedalboard (Spotify) + librosa/scipy; reimplement subset of 85 effects as fallback; ProCraft-Data dataset still covers the full 85-effect design space in tool schemas |
| Slakh soundfont audio sounds synthetic; MOS comparisons unfair | Low | Use CPR/ICA/EDS/FRA as primary metrics; MOS as supplementary; evaluate out-of-distribution on MUSDB18-HQ for realism |

**Fallback plan:** If GRL disentanglement fails, fall back to a pipeline architecture: separate CNN-based timbre encoder (mel-spectrogram statistics), MIDI-derived content encoder, and expression defined residually. Weaker unified-architecture claim but ProCraft-Data dataset contribution and text-conditioned modification pipeline remain intact.

## 12. Stronger Follow-Up

**Full text-to-music synthesis with interpretable factor control:** Extend ProCraft-Model to generation by sampling [TMB] and [EXP] from a learned prior conditioned on text, combined with a content sequence from a text-to-MIDI model. This enables full text-to-music synthesis with explicit factor control (change the generated timbre without regenerating the melody).

**Real-time interactive studio tool:** A DAW plugin where a user describes a modification in natural language, the factor router identifies which factor to adjust, the factor generator proposes the modification, and the user can manually fine-tune individual [TMB] or [EXP] token values for sub-window-level control.
